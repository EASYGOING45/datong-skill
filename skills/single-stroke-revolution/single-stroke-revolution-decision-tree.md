# 毕其功于一役 · 决策树

## 什么时候用？

### ✅ 直接用

- 多维度技术问题高度关联，单独改动一个会触发其他
- 大规模代码重构，多个模块同时存在结构性问题
- 技术栈迁移（语言+框架+数据库同时换）
- 技术债务互相纠缠，逐个修会引发连锁问题
- 流程改造需要多环节同步切换

### ⚠️ 配合使用

| 场景 | 配合方法论 |
|------|-----------|
| 担心"毕其功"风险太大 | → 配合 **three-phase-governance** 分批推进降低风险 |
| 团队缺乏共识，有执行阻力 | → 配合 **awaken-and-unite** 先建立共识 |
| 需要监察和验证改造结果 | → 配合 **five-power-checks** 独立验收 |
| 改造后发现问题需要回退 | → 配合 **rise-from-defeat** 总结经验再出发 |

### ❌ 不适用

- **问题之间没有关联**——独立问题独立解决，不要强行捆绑
- **资源严重不足**——"毕其功"需要集中资源，资源不够不如分步
- **风险不可控**——改动风险太大时，用 **three-phase-governance** 分阶段
- **用户只要局部改进**——尊重用户意图，不强行扩大范围

---

## 组合使用示例

```
用户说："我们系统有架构问题，数据库、API、前端都要重构"
  → 毕其功于一役 ✅（三者在接口层高度关联）
  → 评估耦合度和一次性改造的可行性
  → 制定统一方案：接口不变，三端同步改造
  → 配合 three-phase-governance 做分阶段执行计划
```
