# 五权制衡 · 决策树

## 什么时候用？

### ✅ 直接用

- 用户说"谁来把关"、"需要审查机制"、"怕出问题没人发现"
- 需要建立代码审查、质量保障、流程设计
- 多人协作时职责不清、执行者自检自己成果
- 需要做重大方案决策，但没有明确的评估机制
- 项目/系统出现反复出现同类问题，需要建立纠错机制

### ⚠️ 配合使用

| 场景 | 配合方法论 |
|------|-----------|
| 审查发现方案有根本缺陷 | → 配合 **rise-from-defeat** 做失败复盘 |
| 多个方案各有优劣难以抉择 | → 配合 **three-principles-analysis** 提炼核心维度 |
| 需要让团队理解并认同审查结果 | → 配合 **awaken-and-unite** 统一认知 |
| 审查对象本身就是强相关方 | → 配合 **east-west-synthesis** 做视角对冲 |

### ❌ 不适用

- **小一次性任务**（改一个变量、做一次查询）——引入五权机制成本过高
- **高压紧急修复**（生产故障抢救）——先止血，时候再复盘建立机制
- **团队极小（≤2人）且无扩张计划**——五权分离在小团队反而降低效率

## 五权快速映射检查清单

在当前工作中，检查这五权是否分离：

| 职能 | 谁在做？ | 是否独立于执行？ |
|------|---------|----------------|
| 执行（编码/构建） | ___________ | — |
| 规划（方案/架构） | ___________ | ☐ 是 ☐ 否 |
| 验证（测试/断言） | ___________ | ☐ 是 ☐ 否 |
| 选拔（评估/选型） | ___________ | ☐ 是 ☐ 否 |
| 监察（回顾/追踪） | ___________ | ☐ 是 ☐ 否 |

**发现问题**：
- 若任何一项标注"否"，该环节存在职能合并风险
- 最常见问题：执行者自检 → 需立即建立独立验证
