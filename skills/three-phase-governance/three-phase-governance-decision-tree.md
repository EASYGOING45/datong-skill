# 三阶段推进 · 决策树

## 什么时候用？

### ✅ 直接用

- 推行新技术规范，团队有抵触情绪
- 团队工作方式转型（引入 Code Review、TDD、CI/CD）
- 大型技术迁移需要分阶段交付
- 新工具/新框架需要团队逐步 adoption
- 变革规模大，无法一步到位

### ⚠️ 配合使用

| 场景 | 配合方法论 |
|------|-----------|
| 第一阶段（破旧）阻力太大 | → 配合 **awaken-and-unite** 先统一共识 |
| 第二阶段（教育）效果不好 | → 配合 **aid-the-weak** 照顾弱势学习者 |
| 第二/三阶段遇到失败和反弹 | → 配合 **rise-from-defeat** 克服挫折 |
| 三阶段推进后又回到旧习惯 | → 配合 **single-stroke-revolution** 重新强制切换 |

### ❌ 不适用

- **变革规模很小，不需要分阶段**——直接做就好
- **紧急的一次性切换**——此时用 **single-stroke-revolution** 快速搞定
- **团队已有强烈自发意愿**——可以跳过第一阶段，直接进教育培养
- **每次只推进一件事**——不需要三阶段框架，用基础执行方法

---

## 组合使用示例

```
用户说："我想在团队推行 TDD，但大家都习惯了先写代码再写测试"
  → 三阶段推进 ✅（破旧立新→教育培养→制度自治）
  → 第一阶段：强制要求所有新功能先写测试（破旧）
  → 第二阶段：提供 TDD 培训、lint 规则辅助（立新+教育）
  → 第三阶段：TDD 成为团队默认文化（自治）
  → 配合 five-power-checks 验证推进效果
```
