# 融贯中西 · 决策树

## 什么时候用？

### ✅ 直接用

- 用户说"该用 A 还是 B"、"这两个方案哪个好"、"能不能结合"
- 技术选型时多个框架各有所长，难以取舍
- 架构设计需要在不同风格（如微服务 vs 单体）之间做决策
- 新旧方案并存，需要决定迁移策略
- 团队有不同工作方式，需要统一但不能简单强制一方

### ⚠️ 配合使用

| 场景 | 配合方法论 |
|------|-----------|
| 融合后产生新方案需要验证 | → 配合 **know-hard-act-easy** 先小范围试跑 |
| 融合方案涉及多方利益相关者 | → 配合 **awaken-and-unite** 统一团队认知 |
| 各方方案都有根本性缺陷 | → 配合 **rise-from-defeat** 先做失败复盘 |
| 决策需要多维度评估 | → 配合 **three-principles-analysis** 提炼核心维度 |
| 融合涉及权力/职责分配 | → 配合 **five-power-checks** 建立决策制衡 |

### ❌ 不适用

- **选项完全互斥、无法共存**——此时需要决断（可参考 **power-capability-split** 分离权力）
- **对某方案理解还非常浅**——先深入研究一方，再谈融合
- **时间极其紧迫只能快速选一**——融合成本高，不适合高压场景
- **融合会引入指数级复杂性**——接受"不是所有东西都需要融合"

---

## 融合可行性快速判断

```
遇到 A vs B 的选择：
  → 两者能否在系统中共存？（功能重叠还是互补？）
    → 可以共存 → 考虑融贯 ✅
    → 完全互斥 → 不适合融合，走决断路径 ❌
  → 对 A 和 B 的理解深度是否足够？
    → 不够 → 先各自做一次 **know-hard-act-easy** 分析
    → 够 → 进入融贯步骤
```
